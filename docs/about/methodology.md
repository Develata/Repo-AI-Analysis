# 评分方法与时效

Repo-AI-Analysis 的分数是**带证据边界的结构化判断**，不是跨领域项目的绝对排名。分数适合回答“同类项目各自强弱在哪里”，不应脱离报告日期、证据等级与使用场景单独引用。

## 当前计算契约

每篇报告包含十个 `0–5` 维度；当前 `overall_score` 是十维**等权算术平均**：

```text
overall_score = (r1 + r2 + ... + r10) / 10
```

站点构建会验证所有维度存在、数值有限且落在 `0–5`。推荐度同时参与综合分，因此综合分既包含工程质量，也包含在报告语境下的采用判断。

| 维度 | 主要问题 |
|---|---|
| 功能完整度 | 是否覆盖其声明的核心任务，关键能力是否真实可用？ |
| 易用性 | 安装、配置、日常操作与失败恢复是否清晰？ |
| 性能 | 对目标工作负载是否高效，是否存在明确资源代价？ |
| 代码质量 | 架构、测试、类型、安全边界与维护性如何？ |
| 文档 | 入门、参考、迁移、故障排查与限制是否说明充分？ |
| 社区 | 维护者响应、贡献面、issue/PR 健康度与生态信号如何？ |
| 成熟度 | 项目年龄、发布稳定性、兼容承诺与生产使用信号如何？ |
| 可扩展性 | 插件、API、协议、部署形态与二次开发边界如何？ |
| 安全性 | 默认配置、权限、供应链、漏洞响应与数据边界如何？ |
| 推荐度 | 在报告明确的目标用户和约束下，是否值得采用？ |

## 1–5 分锚点

| 分数 | 解释 |
|---:|---|
| 1 | 核心能力缺失、不可验证或风险显著，不建议采用。 |
| 2 | 可演示但边界明显；需要大量补救工作或只适合实验。 |
| 3 | 基本可用；存在已知权衡，适合匹配场景的审慎采用者。 |
| 4 | 能力和工程质量较强；限制明确，可用于严肃工作。 |
| 5 | 在该维度有充分证据支持的领先表现；不是“无缺点”。 |

中间值表示介于相邻锚点之间。Stars、forks、发布数等只作为信号，不能单独推出质量分。

## 证据等级

报告 frontmatter 中的 `evidence` 描述本次验证边界。可粗略理解为：

1. **文档/元数据审阅**：README、官方文档、GitHub API、release 与治理文件；
2. **源码/本地静态检查**：目录、依赖、测试与关键实现，但未运行；
3. **构建或 smoke test**：实际安装、构建或执行核心路径；
4. **部署/benchmark**：在声明环境中运行并记录可复现实测。

较高等级不自动产生较高分，但会提高结论置信度。报告会明确“未安装”“未部署”“未 benchmark”等负面边界。

## 日期语义

- `updated`：这篇分析文本最近更新的日期；
- `last_checked`：报告引用的仓库状态最近检查日期；
- `last_verified`：若存在，表示更强验证活动的日期，不进入列表索引；
- 仓库自身最近 commit/release 日期属于报告证据，不等于报告核验日期。

Analysis 页将超过 **180 天**未核验的报告标为“待复核”。这只是维护队列阈值，不表示报告已经错误。

## 比较限制

- 优先比较同一末级目录、相似目标用户和相近证据等级的项目；
- 跨目录比较是探索性对照，不是严格排序；
- 雷达图用于观察形状，精确差异应查看分数表和原报告证据；
- 同分不表示可互换：许可、部署条件、平台支持与风险模型可能完全不同。
